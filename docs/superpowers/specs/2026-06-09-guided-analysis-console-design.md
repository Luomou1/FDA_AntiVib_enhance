# FDA Antivib Guided Analysis Console 界面设计

## 1. 目标

在不改变核心算法、参数含义、线程模型、结果模型和导出行为的前提下，重新设计 FDA Antivib 桌面程序的整体界面、排版布局和交互层级。

设计方向采用已确认的 **Guided Analysis Console**：

- 借鉴 Codex 与 Gemini 的清晰导航、任务状态和工具工作台结构。
- 借鉴 Apple iOS/macOS 的克制层次、统一间距和系统主题适配。
- 借鉴 Claude 的温和色彩、低视觉噪声和内容优先表达。
- 面向白光扫描干涉科研分析场景，避免消费级产品的装饰性设计干扰数据判断。

## 2. 已确认决策

- 整体方向：Guided Analysis Console。
- 主题策略：浅色优先，支持跟随系统切换深色主题。
- 信息密度：均衡。
- 参数交互：右侧仅显示当前步骤参数。
- 目标窗口尺寸：`1400 × 900`。
- 最低可用窗口尺寸：`1180 × 760`。

## 3. 信息架构

主窗口由五个稳定区域组成：

1. 顶部工具栏。
2. 左侧全局导航。
3. 左侧分析步骤导航。
4. 中央结果工作区。
5. 右侧当前步骤参数面板。
6. 底部紧凑状态栏。

### 3.1 顶部工具栏

顶部工具栏承担全局任务控制，不再使用大面积标题卡片。

内容包括：

- 应用名称和当前工作流名称。
- 当前数据源或项目路径的简短摘要。
- 分析状态标签。
- 主题模式入口：跟随系统、浅色、深色。
- 自动 K0 主操作。
- 开始分析主操作。

开始分析使用主色按钮；自动 K0 使用次级按钮。分析运行时，两者进入明确的忙碌状态，避免重复提交。

### 3.2 全局导航

最左侧使用约 `44px` 宽的图标导航栏，提供四个一级页面：

- 工作台：参数配置和结果预览。
- 结果：聚焦所有二维、三维、频谱与诊断结果。
- 日志：查看完整运行日志。
- 设置：主题及后续全局偏好。

一级导航只改变主工作区视图，不改变当前分析参数。

### 3.3 分析步骤导航

工作台页面左侧使用约 `180px` 宽的步骤栏：

1. 数据来源。
2. 扫描采样。
3. 算法配置。
4. 运行与导出。

每个步骤显示当前状态：

- 未配置。
- 已配置。
- 需要处理。
- 正在运行。
- 已完成。

步骤栏负责选择右侧参数页面。参数校验失败时，程序自动切换到对应步骤，并聚焦错误控件。

### 3.4 当前步骤参数面板

右侧参数面板宽度约 `300px`，使用 `QStackedWidget` 管理四个步骤页面。任何时刻只显示当前步骤的参数，降低认知负担。

#### 数据来源

- 数据来源类型。
- 图像文件夹路径。
- MAT 文件路径。
- 数据源有效性摘要。

#### 扫描采样

- 起始高度。
- 均匀或非均匀采样模式。
- 步长。
- `scan_log` 路径。
- 位移摘要。

#### 算法配置

- 固定 K0。
- 窗口大小。
- 窗函数。
- 补零策略。
- 有效范围扩展开关。
- 左右扩展帧数。
- 拟合方式。
- 解包裹方式。
- FDA 和 PhaseGap 工作流。

#### 运行与导出

- 当前配置摘要。
- 自动 K0 结果摘要。
- Active range 摘要。
- 开始分析。
- 导出数据。
- 导出图片。

### 3.5 中央结果工作区

中央区域始终获得最大可用空间。结果不再依赖单层横向长标签页，而是按用途分组：

- 核心结果：`h_prime`、`h`、`phi0`、对比图。
- 三维分析：`h` 三维、`h_prime` 三维。
- 频谱：全局 K0 频谱。
- PhaseGap 诊断：仅对应工作流显示现有诊断图层和摘要。

画布工具栏保留重置视角、复制视图和 Matplotlib 导航能力。当前图层、坐标和值在画布附近显示，不占用独立常驻右栏。

结果页在尚未运行分析时显示简洁空状态，引导用户完成参数步骤，而不是显示空白画布。

### 3.6 日志与状态

底部区域缩减为单行紧凑状态栏：

- 当前任务文本。
- 进度条。
- 最新一条日志。
- 展开完整日志的入口。

完整 `QPlainTextEdit` 日志移到一级“日志”页面，避免长期占用垂直空间。

## 4. 视觉系统

### 4.1 色彩

浅色主题：

- 应用背景：冷灰白。
- 面板背景：白色或轻微灰白。
- 主色：克制蓝紫色。
- 信息提示：暖橙色。
- 成功状态：低饱和绿色。
- 错误状态：低饱和红色。

深色主题：

- 使用中性深灰而不是纯黑。
- 保持数据画布和界面容器之间的层级差。
- 不修改科学图表的色图含义。

### 4.2 字体

- 中文界面优先使用 `Microsoft YaHei UI`。
- Windows 英文界面回退到 `Segoe UI Variable` 或 `Segoe UI`。
- 数字、K0、范围和坐标读数使用系统等宽字体。
- 不引入网络字体或新的运行时依赖。

### 4.3 尺寸与间距

- 常规控件高度约 `32px`。
- 主按钮高度约 `36px`。
- 面板圆角约 `10px`。
- 控件圆角约 `7px`。
- 基础间距使用 `4 / 8 / 12 / 16 / 24` 序列。
- 使用弱边框和背景层级，不依赖厚重阴影。

### 4.4 动效

仅在以下位置使用短时反馈：

- 页面或步骤切换。
- 状态标签变化。
- 分析进度。
- 按钮悬停与按下。

目标时长为 `120–180ms`。不使用会影响绘图性能或科学判断的持续动画。

## 5. 主题系统

主题模式包含：

- 跟随系统。
- 浅色。
- 深色。

默认模式为跟随系统，并在系统主题无法可靠识别时回退到浅色。

主题由独立样式模块生成应用级 QSS。主窗口和辅助窗口复用同一套调色板，避免在控件创建处散落 `setStyleSheet()`。

Matplotlib 和 PyVista 画布只调整容器、工具栏和文字背景；数据本身的色图和数值范围保持原行为。

## 6. 代码结构

本次重构保持现有业务控制器和公开字段兼容，优先拆分纯 GUI 结构：

- `app/gui/theme.py`：主题令牌、系统主题检测和 QSS 生成。
- `app/gui/widgets/`：导航按钮、步骤按钮、状态标签、参数页面标题等复用控件。
- `app/gui/main_window.py`：保留业务信号、线程调度、结果刷新和导出协调。

四个参数页面可以由主窗口中的独立构建方法创建。若实施时单文件复杂度仍过高，再提取为独立页面模块，但不移动业务状态所有权。

现有控件字段名和信号连接应尽量保持，例如：

- `data_source`
- `folder_edit`
- `sampling_mode`
- `fixed_k0`
- `phase_gap_method`
- `auto_k0_button`
- `start_button`
- `tabs`
- 各结果画布字段

这样可以控制重构范围，并降低对现有回调和测试的影响。

## 7. 状态与错误处理

- 参数错误必须映射到具体步骤。
- 运行期间锁定所有会改变分析输入的参数控件。
- 结果浏览、复制视图和日志查看在运行期间保持可用。
- 分析失败后恢复参数和运行按钮，并在状态栏、日志页和错误对话框中保持一致信息。
- 自动 K0 与正式分析使用不同的运行状态文本。
- 无数据、无结果和工作流不支持某诊断层时使用明确空状态。

## 8. 兼容性与边界

本次包含：

- 主窗口整体布局重构。
- 应用级浅色和深色主题。
- 参数步骤交互。
- 结果页面分组。
- 日志页面与紧凑状态栏。
- 主窗口相关 GUI 测试。

本次不包含：

- FDA、PhaseGap 或 NUFFT 算法修改。
- `AnalysisParams`、`AnalysisResult` 等共享数据结构修改。
- 数据库、持久化或项目文件格式。
- 新增第三方依赖。
- 绘图算法和科学色图重新定义。

## 9. 验证标准

### 9.1 自动验证

- 主窗口可在无数据状态下构建。
- 最低尺寸下所有一级导航、步骤导航和主操作可访问。
- 四个参数步骤可以切换，原控件值不丢失。
- 数据来源、采样模式、工作流和范围扩展联动行为保持一致。
- 浅色、深色和跟随系统模式可切换。
- 分析运行状态正确锁定和恢复参数。
- 自动 K0 完成后摘要和频谱刷新。
- 分析完成后核心结果与对应诊断页刷新。
- 导出按钮仍按结果状态启用。

### 9.2 手工验证

- 在 `1400 × 900` 下检查均衡密度布局。
- 在 `1180 × 760` 下检查可用性和滚动行为。
- 检查 Windows 常见缩放比例下的文本截断。
- 检查浅色和深色主题中的对比度、禁用态和焦点态。
- 检查 Matplotlib 与 PyVista 工具栏、复制视图、点击读数和弹出分析窗口。

## 10. 完成条件

- 所有现有分析入口和结果入口仍可访问。
- GUI 构建和定向测试通过。
- 现有核心算法测试无回归。
- 主窗口不再使用旧三栏参数堆叠布局。
- 主题和布局样式集中管理。
- 未修改核心算法与共享数据契约。
